# Contributing

Thanks for your interest in improving the SE Harness! Contributions of all kinds are
welcome — bug reports, new agents/skills, docs fixes, and support for additional
stacks or ecosystems. This project is MIT-licensed; by contributing you agree your
contributions are licensed under the same terms.

## Repo layout — read this first

This repository is a **dual-ecosystem plugin marketplace** with one source of truth
and two generated artifacts. Editing the wrong file is the most common mistake:

| Path | Role |
|---|---|
| `plugins/se-harness/` | **Source of truth.** The Claude Code plugin — all changes to agents, skills, commands, hooks, and scripts happen here. |
| `plugins/se-harness-copilot/` | **Generated — never edit by hand.** Built from `se-harness` by `plugins/se-harness/scripts/build-copilot-plugin.sh`. Hand edits are destroyed on the next build (`rm -rf`). |
| `.claude-plugin/marketplace.json` | Marketplace manifest, read by both Claude Code and Copilot CLI. Edit directly. |
| `.github/plugin/marketplace.json` | **Generated mirror** of the above (Copilot's canonical location). Regenerated by the build script. |
| `templates/` | Files the harness copies into user projects (`AGENTS.md.tmpl`, `workspace.yaml`, hooks, memory seeds). |
| `registry/recommendations.json` | Stack-detection → companion-tooling recommendations used by `/harness-bootstrap`. |
| `docs/` | Setup guides (Claude + Copilot) and per-platform notes. |

## Development workflow

1. **Fork and branch.** Create a topic branch off `main`.
2. **Make your change in `plugins/se-harness/`** (or docs/templates/registry as
   appropriate).
3. **Regenerate the Copilot variant** if you touched anything under
   `plugins/se-harness/`:

   ```bash
   bash plugins/se-harness/scripts/build-copilot-plugin.sh
   ```

   Commit the regenerated `plugins/se-harness-copilot/` and
   `.github/plugin/marketplace.json` together with your source change.
4. **Validate** (required — CI for the community plugin directory runs the same
   check):

   ```bash
   claude plugin validate . --strict
   claude plugin validate ./plugins/se-harness --strict
   ```

5. **Test locally** without installing:

   ```bash
   claude --plugin-dir ./plugins/se-harness
   ```

   Then exercise what you changed — e.g. `/se-harness:harness-init` in a scratch
   project, or trigger the hook path you modified. `/reload-plugins` picks up edits
   without restarting.
6. **Open a pull request** against `main`. Describe what you changed and how you
   tested it. Keep PRs focused — one logical change per PR.

## Versioning and releases

**Do not bump versions in a contribution PR** — maintainers handle releases.
Versions are pinned, so users only receive updates when the version string changes.
A release bumps three places in lockstep (semver):

- `plugins/se-harness/.claude-plugin/plugin.json` → `version`
- `.claude-plugin/marketplace.json` → `metadata.version`
- `plugins/se-harness/scripts/build-copilot-plugin.sh` → the `version` in the
  generated `plugin.json` heredoc

followed by a rebuild, a `vX.Y.Z` tag, and a GitHub release.

## Conventions

- **Agents** (`plugins/se-harness/agents/*.md`): YAML frontmatter with `name`,
  `description`, and optional `tools`. The build script converts these to Copilot's
  `NAME.agent.md` format — keep frontmatter fields on single lines so its parser
  handles them.
- **Skills** (`plugins/se-harness/skills/<name>/SKILL.md`): shared verbatim across
  both ecosystems — write them platform-neutral.
- **Commands** (`plugins/se-harness/commands/*.md`): reference scripts as
  `${CLAUDE_PLUGIN_ROOT}/scripts/...`; the build script rewrites these paths for
  Copilot. Don't hardcode absolute paths.
- **Scripts** (`plugins/se-harness/scripts/*.sh`): POSIX-leaning bash, `set -eu`,
  no network calls. The plugins must keep working fully offline — see
  [PRIVACY.md](./PRIVACY.md); a change that adds telemetry or remote calls will not
  be accepted without a privacy-policy update and very good reason.
- Line endings: files are committed with LF; a CRLF-converting editor on Windows is
  fine (git handles it), but don't commit mass line-ending churn.

## Reporting bugs and proposing features

Open a [GitHub issue](https://github.com/rbhattarai/se-harness/issues)
with reproduction steps (for bugs: OS, Claude Code / Copilot CLI version, and the
command or hook involved). For larger features — new agents, new ecosystem targets,
changes to the goal-loop gates — please open an issue to discuss before investing in
a PR.
