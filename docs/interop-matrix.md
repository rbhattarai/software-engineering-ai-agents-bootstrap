# se-harness interop matrix: Claude Code × GitHub Copilot (2026-07-16)

How each framework component ships on each platform. Details: `docs/claude/platform-notes.md`
and `docs/copilot/platform-notes.md`.

| Component | Claude Code | GitHub Copilot |
|---|---|---|
| Marketplace | `.claude-plugin/marketplace.json` — `/plugin marketplace add owner/repo` | **Same file** (Copilot CLI reads `.claude-plugin/` too) + generated mirror at `.github/plugin/marketplace.json` (canonical location; rebuilt by `build-copilot-plugin.sh`, never hand-edited) |
| Plugin | `plugins/se-harness` (`.claude-plugin/plugin.json`) | `plugins/se-harness-copilot` (generated by `scripts/build-copilot-plugin.sh`; `plugin.json` at root) |
| Commands / goal loop | `commands/*.md` → `/harness-*` | plugin `commands/` **and** each command emitted as a skill (`skills/harness-*/SKILL.md` — Copilot commands are skill-shaped, invoked `copilot harness-goal …`) + `.github/prompts/*.prompt.md` for VS Code Chat; script paths rewritten to `tools/harness/` |
| Agents (11-role roster) | `agents/*.md` (frontmatter incl. `tools`) | plugin `agents/*.agent.md` (tools moved to body guidance) + repo-level `.github/agents/*.md` via `export-agents.sh copilot` for the coding agent |
| Skills (7 harness + 6 command-derived) | `skills/<name>/SKILL.md` | **Same SKILL.md standard** — copied verbatim into the Copilot plugin |
| Hooks: gate-check, contract-check (blocking) | `hooks/hooks.json` PreToolUse, exit 2 blocks (deny-JSON also works) | bundled in the plugin (`hooks.json`, plugin-root-relative paths) AND/OR repo-level `.github/hooks/se-harness.json`, via `copilot-hook-adapter.sh` (exit 2 → **deny JSON — the one blocking mechanism valid on every surface**). VS Code uses Claude semantics (exit 2 blocks, PascalCase events, reads `.claude/settings.json` + plugin `hooks/hooks.json`) |
| Hooks: org-validate, memory-log (feedback) | PostToolUse | same adapter, `post` mode (never fails closed) |
| Hooks: inject-context | UserPromptSubmit stdout → context | `userPromptSubmitted` (CLI/agent) and `UserPromptSubmit` (VS Code) exist — context-injection effect still **unverified**; not wired yet |
| MCP (github, atlassian, code-graph) | `.mcp.json` (plugin or project) | `.vscode/mcp.json` / plugin `.mcp.json` / enterprise-managed |
| Instructions | CLAUDE.md (+AGENTS.md) | AGENTS.md native; **reads CLAUDE.md directly**; thin `.github/copilot-instructions.md` pointer |
| `.harness/` artifacts (profile, REQs, memory, lock) | plain files | plain files — fully shared |
| Enterprise distribution | `managed` plugin scope / managed settings | enterprise-managed plugins (auto-install, forced hooks/MCP) |

**Design rule (unchanged from B4/ECC):** the durable behavior lives once — commands, agents,
skills, scripts, templates in the Claude-first plugin — and platform adapters stay thin
(`build-copilot-plugin.sh`, `copilot-hook-adapter.sh`, `export-agents.sh`). If supporting a
platform requires editing workflow *content*, the content is in the wrong place.

**Not yet ported to Copilot:** context-injection hook (see matrix row), Claude-only hook events
(worktree, subagent, task events), skills auto-activation parity (Copilot skill triggering is
name/description-based too, but activation behavior differs — observe in practice).
